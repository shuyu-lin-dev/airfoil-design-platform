1. 大模型 API Key 在本项目中的并行调用策略

核心结论

本项目不需要为每个最终用户创建独立的 API Key。 推荐架构：1 把服务端 Key → 1 个 LLM Proxy/Gateway → N 个用户。 服务端持有真实 Key，用户请求通过后端代理转发，由网关层负责鉴权、限流、配额和审计。 若未来用量增长或需多模型切换，引入 LiteLLM Proxy 即可在不改业务代码的前提下横向扩展。

三种方案对比

方案做法适用场景本项目适合吗？
A. 单 Key 共享 后端持有一把 Key，所有用户请求通过后端转发 MVP、小团队、内部工具 MVP 首推
B. 多 Key 分发 为每个用户/团队创建独立子 Key 多租户 SaaS、需按用户计费 远期可选
C. LLM Proxy 网关 部署 LiteLLM/Portkey，生成虚拟 Key 给用户 生产环境、多模型、多租户 生产推荐

MVP 阶段的理由

  • 第一版没有用户登录和权限系统，不存在"多用户"概念
  • 实际使用者是后端调用者、测试人员、未来前端，本质上都是内部调用者
  • DeepSeek API 无硬性速率限制（官方声明不约束用户 rate limit），MVP 用量远不会触及阈值
  • 架构最简单，Key 通过环境变量注入，config/settings.py 统一管理
  • 后续升级到方案 C 时，业务代码无需改动——只需在 proxy 层配置

Key 的安全管理

  • Key 存入 .env 文件，通过 python-dotenv 加载
  • .env 加入 .gitignore，不提交到版本控制
  • 提供 .env.example 模板文件（含空值占位）
  • CI/CD 中通过 GitHub Secrets 或环境变量注入
  • 代码中仅通过 config/settings.py 读取，不在 service/core 中硬编码

生产环境升级路径（方案 C）

当以下任一条件触发时，将方案 A 升级为方案 C（LiteLLM Proxy）：

触发条件

  • 引入用户登录系统，需要按用户隔离调用额度
  • 需要同时使用多个模型（DeepSeek + GPT + Claude 等）做对照
  • 单个 Key 的用量接近平台软限制
  • 需要审计每次 LLM 调用的用户、模型、token 消耗和成本

LiteLLM Proxy 提供的能力

  • 虚拟 Key 生成——每个用户一把虚拟 Key，映射到同一把真实 Key
  • 按用户/团队的 RPM、TPM、预算上限
  • 统一日志（user_id, model, tokens, cost）
  • 多模型 fallback 路由（DeepSeek 挂了自动切 GPT）
  • Key 轮换不触及应用代码

多用户并发：一把 Key 会互相拖慢吗？

简短回答：不会互相拖慢。一把 Key 不等于"一次只能处理一个请求"。

类比：餐厅厨房

一把 API Key 就像一家餐厅的厨房，而不是一个灶台。

多个用户同时点菜（发请求），厨房会同时开多个灶台并行烹饪。 只要厨房的灶台数（并发连接数）够用，每个用户的菜（响应）不会因为别人在点菜而变慢。

每个请求是独立的 HTTP 连接，不排队、不互斥。用户 A 的请求处理和用户 B 的请求处理在服务端是并行的。

真正影响速度的因素

总并发连接数超过平台限制后，新请求需排队等待。DeepSeek 免费版约 5 并发，专业版约 50。
服务端负载平台整体繁忙时（如高峰期），每个请求的推理时间可能变长——但这和 Key 无关，是服务端容量问题。
Token 消耗速率输出 token 越多，响应时间越长。这是模型推理速度决定的，不是 Key 争抢导致的。
你的代码如果用 await 异步发请求，多个请求并发执行，总耗时 ≈ 最慢那个。如果用同步 for 循环串行调用，才会一个等一个。
结论：一把 Key 多人用，就像多人共用一个 Wi-Fi——每个人该多快还是多快，除非总带宽（并发上限）被打满。 MVP 阶段用户量极少，完全不用担心。未来用户量增长后，升级 LiteLLM Proxy 即可在网关层按用户分配配额，防止单用户占用过多并发。

并行调用可行性分析

并行场景是否可行注意事项
单用户同时调用多个工具
（如：同时做气动预测 + 强度预测 + 优化）		 可行 FastAPI 本身支持异步并发（async def + asyncio.gather）。多个 LLM 调用互不阻塞。只需确保 Key 的并发连接数充足。
多用户同时调用同一工具 可行 同一把 Key 天然支持多并发请求。DeepSeek 免费版约 5 个并发连接，专业版约 50 个，企业版 50+ QPS。MVP 阶段无用户系统，实际并发可忽略。
同一请求内多次 LLM 调用并行
（如：ReAct 循环中同时查询多个知识源）		 可行 使用 asyncio.gather 或 concurrent.futures.ThreadPoolExecutor 即可。注意控制最大并发数，避免触发服务端动态限流（429）。
长时间运行的优化循环中频繁调用 LLM 需注意 第一版优化循环为占位实现，不调用 LLM。未来若引入 LLM 驱动的优化，需在 proxy 层做 rate limiting + 指数退避重试。

2. 提示词工程范围

前置说明

第一版 MVP 不实现智能体（见 spec 第 4 节"第一版不做"），所有 API 后端是占位函数，不调 LLM。 但提示词工程的设计应前置，避免后期智能体接入时返工。以下定义的是智能体阶段需要覆盖的 Prompt 类型、模板结构和工程边界。

System Prompt 推荐结构

# 角色
你是一个翼型气动与强度设计助手，服务于学生和工程师用户。
你可以帮助用户生成翼型、预测气动/强度性能、执行优化设计，
以及解释翼型设计原理。

# 可用工具
- generate_airfoil_2d: 输入 12 个 CST 参数，生成 200 个二维翼型点
- predict_aerodynamics: 输入 CST 参数和工况(mach, aoa)，预测升阻比、Cp、压力场、速度场
- predict_structure: 输入 CST + 工况 + 结构 + 材料，预测应力和重量
- optimize_aerodynamic: 气动优化
- optimize_structural: 强度优化
- optimize_coupled: 耦合优化
- generate_teaching_airfoil: 通过中弧线和厚度控制点生成翼型
- inverse_cst: 从翼型坐标反推 CST 参数

# 行为准则
- 所有预测/优化结果为占位数据(is_stub=true)，请在回答中如实告知用户
- 对于学生用户，用通俗语言解释专业术语
- 对于工程师用户，可以简略给出关键数值
- 如果用户请求超出你的能力范围，诚实说明而不是猜测

# 领域知识
- CST 参数前 6 个为上表面系数，后 6 个为下表面系数
- 工况参数中 mach 无量纲，angle_of_attack 单位为 degree
- 重量统一使用 weight(N)，不是 mass(kg)
- 坐标系：x=弦向，y=厚度方向，z=展向

提示词工程边界

MVP 阶段可做（为智能体铺路）

  • 在 config/ 下新建 prompts/ 目录，存放 prompt 模板文件
  • 定义 System Prompt 的初版文本
  • 每个 API 的输入输出 schema 就是天然的 "tool description"
  • 在 contracts 中为每个 API 增加 description 字段，供未来智能体读取

MVP 阶段不做

  • 不写 ReAct 循环逻辑
  • 不写参数提取 prompt
  • 不做 RAG 检索链路
  • 不调 DeepSeek API
  • 不做对话历史管理
  • 不做 streaming 响应

3. 本地知识库构成

知识库用于 RAG（检索增强生成）：当用户提出专业问题时，从知识库检索相关文档片段注入 Prompt， 让 LLM 基于权威资料回答，而非依赖模型自身可能过时或不准确的训练记忆。 参考 AirfoilRAG (2025)、OpenFOAMGPT (2025) 和 ChatCFD (2025) 的实践。

重点参考：AirfoilRAG 框架可借鉴之处

AirfoilRAG (Fan et al., 2025, NPU) 是首个将 LLM+RAG 用于翼型气动设计的框架，与本项目目标高度吻合。 以下为本项目可直接借鉴的设计决策：

AirfoilRAG 做法本项目可借鉴
使用 AFBench 数据集（10,000 个 NACA 翼型）作为知识库 将 AFBench 纳入本地知识库 C 类（气动数据），提供翼型-气动性能的配对参考
混合检索：几何特征 + 气动特性，两阶段（粗排→精排） 智能体阶段实现翼型检索时采用类似策略：先按几何相似度粗筛，再按气动指标精排
使用 Parsec 参数化 + XFOIL 计算工具 本项目使用 CST 参数化（12 参数），可与 Parsec 互补。XFOIL 可作为气动预测的校验参考
气动知识蒸馏：微调 LLM 预测气动性能 远期可考虑。MVP 阶段用占位函数 + is_stub 标记，未来接入真实 PyTorch 模型时做对比评估
整体预测精度 MAPE ≤ 5% 作为本项目未来气动预测模型的精度 benchmark 参考
