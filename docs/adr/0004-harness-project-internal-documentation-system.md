# ADR 0004: Harness 架构——项目内文档系统

> 日期：2026-05-27（原始），2026-05-30（从 DECISIONS.md 提炼为 ADR）
> 状态：accepted

## 背景

项目处于后端 MVP 起步阶段，最缺的是 agent 可读的事实源、任务边界、验证命令和跨会话状态。需要一个轻量的 harness 系统来约束 agent 行为。

## 决策

**第一版采用项目内 harness**，而非先创建通用 harness 再反向生成。具体结构：

1. **入口路由**：`AGENTS.md` 作为 agent 入口，只做索引路由，不承载详细规则、标准命令或领域硬约束。
2. **规则分层**：规则拆分到 `.harness/instruction/rules/`，按工作流、领域、测试、artifact 和架构边界分文件维护，便于 agent 按任务类型读取相关约束。
3. **领域语境**：`CONTEXT.md` 保留在仓库根目录（可发现性最高），记录领域语言、字段边界和已解决歧义。
4. **规格分层**：完整规格 `docs/backend-mvp-full-spec.md`（英文文件名，权威细节源）+ `docs/spec.md`（摘要）+ `docs/api-contracts.md`（契约摘要），摘要与完整规格冲突时以完整规格为准。

## 原因

- 项目内 harness 更贴近当前需求，能马上约束后端 tracer bullet。
- 通用 harness 过早抽象会把注意力从业务契约和验证闭环上移走。
- 规则分文件能降低冷启动阅读成本，细节渐进披露。
- 根目录文件保持稀缺：只放入口类和状态类文件。

## 约束

- `AGENTS.md` 不承载详细规则、标准命令或领域硬约束，只保留索引。
- 规则变更必须同步更新 `DECISIONS.md` 和 `PROGRESS.md`。
- 摘要与完整规格冲突时，先暂停并指出冲突。

## 否决方案

- 先创建独立 harness 项目再反向生成——过早抽象。
- 所有规则集中在 `AGENTS.md`——入口文件过重，不符合渐进披露原则。
- `CONTEXT.md` 移入 `docs/`——降低可发现性。
- 删除完整规格只保留摘要——丢失原始意图和细节约束。
