课程的核心判断是：可靠 agent 不是只靠更强模型，而是靠指令、工具、环境、状态、反馈、验证和交接组成的闭环工作系统。

学习路径

讲义、项目、资料库

先理解失败模式和 harness 概念，再用项目练习搭建可读、可验证、可交接的 agent 工作环境，最后把模板迁移到自己的仓库。

核心机制

目标、初始化、执行、验证、交接

从 `AGENTS.md` 明确目标，通过初始化检查建立前提，让 agent 执行任务，再用测试、QA 和进度文件形成反馈闭环。

学生产出

一套能落地的 Harness

学生最终应能给一个真实项目补齐入口文件、进度状态、功能清单、验证命令、观测工件和退出检查。

明确目标把任务、边界和验证标准写进仓库入口。
初始化检查确认依赖、启动命令、示例测试和任务清单。
运行任务一次只激活一个功能项，围绕完成证据推进。
运行反馈用日志、错误消息、健康检查指导修复。
验证评审静态、行为、系统和交接层都通过才完成。
清理交接记录进度，清理临时工件，留下可接手状态。

失败与 Harness：L01-L02

先把失败从“模型不行”改写成“哪一层 harness 缺失”。这一步决定学生后面是换模型、加提示词，还是系统性修补工作环境。

操作卡片

失败先归因，不先换模型

每次 agent 失败后，把问题归到任务规范、上下文供给、执行环境、验证反馈或状态管理，而不是笼统地判定模型能力不足。

诊断循环 五子系统 验证缺口
检查清单
  • 任务是否有显式完成定义，而不是一句宽泛需求？
  • agent 是否能找到技术栈、架构约束和验证命令？
  • 环境能否启动，测试能否运行，失败信息能否指导修复？
  • 跨会话状态是否写进仓库，而不是停在聊天记录里？
应用方式

等模型对照实验

保持模型和任务不变，逐个补齐 `AGENTS.md`、验证命令、进度文件等组件，观察成功率和重建成本变化，定位最该加强的子系统。

执行让 agent 做一个真实任务，记录输出和失败点。
观察失败不要先改 prompt，先看是哪种结构性缺口。
定位层级任务、上下文、环境、反馈、状态五层归因。
修补 Harness补文件、命令、测试、日志或状态工件。
重新执行同模型重跑，用结果衡量改进是否有效。

第一讲. 模型能力强，不等于执行可靠

痛点

真实项目有隐性约束、环境差异、没有现成测试和长会话记忆损耗；模型会写代码，不等于会可靠交付。

概念

能力鸿沟、Harness、Harness 诱导失败、验证缺口、诊断循环、完成定义。

做法

给每个任务写可执行完成标准；创建 `AGENTS.md`；把失败归到五层防御；记录任务成功率和失败层级。

案例

课程材料中的案例显示，同一模型在裸环境和完整 harness 中表现差异巨大；另一个 FastAPI 团队通过入口文件、验证命令和决策记录显著提高成功率。

练习

做一次“无 harness vs 最小 harness”对比实验，并测量 agent 声称完成但独立验证失败的比例。

来源

raw/ai-workflows/lecture-01-model-capability-vs-reliability.md

第二讲. Harness 到底是什么

痛点

很多团队把 harness 误解成一个 prompt 文件，缺少工具、环境、状态和反馈，导致 agent 把时间花在猜项目上。

概念

指令、工具、环境、状态、反馈五子系统；仓库是唯一事实来源；给地图，不给百科全书；约束而非微操。

做法

用 `AGENTS.md` 说明项目、命令和硬约束；开放最小必要工具；锁定运行时；用 `PROGRESS.md` 跟踪长任务；在入口写清验证命令。

案例

课程材料中的 TypeScript + React 团队从空仓库描述逐步增加入口文件、验证命令和状态文件，成功率随 harness 完整度提升。

练习

按五元组给自己的项目打分，最低分子系统先花 30 分钟补强，再用同一任务观察表现变化。

来源

raw/ai-workflows/lecture-02-what-a-harness-is.md

仓库事实源：L03

不在仓库里的知识，对 agent 来说就是不存在。仓库事实源模块把人脑、Slack、Confluence 和 Jira 中的关键约束迁移到 agent 可访问的位置。

来源：lecture-03-repo-as-system-of-record.md
操作卡片

做一次冷启动测试

开新会话，只给仓库内容，检查 agent 能否回答：这是什么系统、如何组织、如何启动、如何验证、当前做到哪里。

知识可见性 System of Record 发现成本
仓库布局
  • `AGENTS.md`：入口、命令、全局硬约束。
  • 模块旁文档：`ARCHITECTURE.md`、`CONSTRAINTS.md`、`TESTING.md`。
  • `PROGRESS.md`：当前进度、阻塞、下一步。
  • `Makefile` 或 scripts：标准化操作路径。
风险雷达

过时文档比没有文档更危险

知识要靠近代码，并和代码一起更新。否则 agent 会遵循错误地图，错误还会显得很自信。

收集隐性知识团队约定、架构决策、运行命令、验证标准。
放进仓库入口文件放高频信息，细节靠近代码。
降低发现成本路径短、标题清楚、按任务主题组织。
冷启动测试用五个问题验证新会话能否接手。
随代码维护文档和变更同审查、同提交、同验证。
讲义详情：仓库即事实源

第三讲. 让代码仓库成为唯一的事实来源

痛点

项目知识散落在仓库外，agent 无法访问，导致每次任务都要猜架构、命令、约束和当前进度。

概念

知识可见性缺口、系统记录、冷启动测试、发现成本、知识衰减率、ACID 类比。

做法

把关键知识写入仓库；入口文件标准化；模块规则靠近代码；用原子提交、一致性验证、并发隔离和持久状态管理 agent 工作。

案例

课程材料中的微服务团队把全局硬约束、服务架构和进度文件写入仓库后，同一 agent 冷启动时能回答关键问题，人工干预减少。

练习

列出项目关键决策，标注仓库内/仓库外，计算知识可见性缺口，并把冷启动五问答不上来的部分补齐。

来源

raw/ai-workflows/lecture-03-repo-as-system-of-record.md

指令路由：L04

`AGENTS.md` 是路由器，不是百科全书。模块目标是让学生会拆分入口指令，避免“每次失败就加一条规则”的指令膨胀。

来源：lecture-04-progressive-disclosure-instructions.md
操作卡片

入口文件 50-200 行

只放项目概览、首次运行命令、全局硬约束和专题文档链接。安全、数据库、测试、部署等细节按需披露。

渐进式披露 中间迷失 信噪比
拆分规则
  • 硬约束放入口顶部，避免埋在长文件中部。
  • 专题文档按任务触发条件命名。
  • 历史教训优先转成测试或自动检查。
  • 每条规则说明来源、适用条件和过期条件。
常见反模式

巨型指令文件

600 行入口文件会吃掉上下文预算、降低当前任务信噪比，并让关键安全约束被“中间迷失”稀释。

审计指令按任务类型标注每条规则是否相关。
裁剪入口保留高频、全局、不可违反的信息。
拆到专题API、数据库、测试、部署分别建文档。
转成检查重复审查意见优先变成 lint、test 或脚本。
讲义详情：指令文件架构

第四讲. 把指令拆分到不同文件里

痛点

每次失败都往入口文件加规则，最终上下文预算被低价值指令占用，关键约束被埋在文件中部，甚至互相矛盾。

概念

指令膨胀、中间迷失效应、指令信噪比、路由文件、渐进式披露、优先级模糊度。

做法

入口文件只保留概览、命令、少量硬约束和专题链接；专题文档 50-150 行；历史笔记转成测试或删除。

案例

课程材料中的 SaaS 团队把 600 行入口文件拆成 80 行路由文件和多个专题文档后，安全约束遵循率和任务成功率明显改善。

练习

给入口文件做信噪比审计，选择 5 类常见任务，找出大多数任务都用不上的规则并移到专题文档。

来源

raw/ai-workflows/lecture-04-progressive-disclosure-instructions.md

连续性与初始化：L05-L06

长任务需要“会失忆的工程师”工作法：开工前先打地基，收工前写日记，让新会话能从仓库工件恢复，而不是重新侦探一遍。

来源：lecture-05-long-running-task-continuity.md；lecture-06-agent-initialization-phase.md
操作卡片

跨会话状态必须持久化

`PROGRESS.md` 记录当前状态、已完成、进行中、已知问题和下一步；`DECISIONS.md` 记录关键选择及其原因。

连续性工件 自举契约 上下文焦虑
初始化验收
  • 项目能从零安装并启动。
  • 至少一个示例测试通过。
  • 自举契约写清命令、状态和结构。
  • 任务分解有明确验收标准。
  • 初始化结束有干净检查点。
关键指标

重建成本

新会话恢复到可执行状态所需时间。好的 harness 目标是几分钟内完成重建，而不是每次重新探索 15-30 分钟。

环境可运行依赖、运行时和服务路径清楚。
示例测试先证明测试框架本身有效。
自举契约启动、测试、项目结构和当前状态。
任务清单每项任务有验收标准。
进度决策会话结束写回状态和为什么。
检查点用提交或明确快照留下可恢复状态。

第五讲. 让跨会话的任务保持上下文连续

痛点

长任务跨会话后，agent 不知道上次做了什么、为什么这样做、测试状态如何，只能重新探索并可能产生漂移。

概念

上下文窗口、连续性工件、重建成本、漂移、上下文焦虑、压缩 vs 重置。

做法

维护 `PROGRESS.md`、`DECISIONS.md`、验证记录和 git 检查点；会话开始读状态，会话结束写回状态。

案例

课程材料中的博客系统对比显示，使用进度、决策、验证和检查点后，新会话重建成本和重复实现明显下降。

练习

设计最小交接模板：仓库状态、运行时状态、阻塞项、下一步行动，并让全新会话只凭模板恢复。

来源

raw/ai-workflows/lecture-05-long-running-task-continuity.md

第六讲. 让 agent 每次工作前先初始化

痛点

agent 一上来写功能，做到一半才发现环境、测试和项目结构没准备好，导致未验证代码累积。

概念

初始化阶段、自举契约、冷启动 vs 热启动、交接就绪性、首次验证时间、下游可用性。

做法

第一个会话只建立基础设施：可运行环境、示例测试、自举契约、任务分解和干净 checkpoint；后续会话再实现业务。

案例

课程材料中的 React 项目显示，独立初始化虽然前期多花时间，但后续会话能快速从任务列表开始，减少跨会话重建成本。

练习

为项目写自举契约，让新 agent 只看仓库启动项目、跑测试、理解进度，记录卡点并补齐契约。

来源

raw/ai-workflows/lecture-06-agent-initialization-phase.md

任务控制：L07-L08

agent 容易 overreach 和 under-finish：同时做很多事，却没有一个行为真正通过。任务控制模块把范围、状态和完成证据外部化。

来源：lecture-07-task-boundaries.md；lecture-08-feature-lists-as-harness-primitives.md
操作卡片

WIP=1 是默认安全设置

任何时刻只允许一个功能项处于 `active`。当前项的验证命令通过后，才能解锁下一个任务。

WIP 限制 功能清单 完成证据
功能项三元组
  • 行为描述：外部可观察行为。
  • 验证命令：如何证明行为成立。
  • 当前状态：`not_started`、`active`、`blocked`、`passing`。
  • 证据引用：commit、日志或测试输出。
状态规则

agent 不能自己标 passing

功能状态应由 harness 根据验证结果推进。agent 可以请求验证，但不能用“看起来完成”替代通过状态门控。

功能队列读取功能清单中的 `not_started`。
只选一个把当前任务置为 `active`。
实现行为不顺手改无关功能或重构。
运行验证执行该功能自己的验证命令。
状态转移通过变 `passing`，失败继续修，依赖缺失变 `blocked`。
讲义详情：任务边界与功能清单

第七讲. 给 agent 划清每次任务的边界

痛点

宽泛任务会让 agent 同时改数据库、路由、前端和中间件，代码很多但端到端没有一个功能跑通。

概念

过度延伸、 不足完成、WIP 限制、完成证据、范围表面、完成压力。

做法

强制 WIP=1；每项任务定义可执行完成证据；范围表面外部化；用验证完成率监控是否允许启动新任务。

案例

课程材料中的 REST API 对比显示，无约束时多个功能半成品，WIP=1 时每个会话代码更少但有效完成率更高。

练习

把“实现用户管理系统”拆成至少 5 个原子工作单元，每个写行为、验证命令和依赖关系。

来源

raw/ai-workflows/lecture-07-task-boundaries.md

第八讲. 用功能清单约束 agent 该做什么

痛点

没有功能清单时，agent 用自己的隐式标准判断“做完”，新会话也无法知道“基本完成”到底意味着什么。

概念

功能清单是 harness 原语、三元组结构、状态机模型、通过状态门控、单一权威来源、反向压力。

做法

用 Markdown 或 JSON 记录功能项；调度器选任务，验证器决定状态，交接器从清单生成摘要。

案例

课程材料中的电商平台从非结构化进度笔记改为结构化功能清单后，新会话能几分钟内知道哪些功能 passing、active 或 not_started。

练习

定义最小 feature list schema，并用它描述 5 个真实功能；对比宽松验证和严格验证的假阳性率。

来源

raw/ai-workflows/lecture-08-feature-lists-as-harness-primitives.md

验证与观测：L09-L11

完成不能由 agent 的信心判定。验证与观测模块把终止标准、端到端测试、错误反馈、运行时信号和过程工件都纳入 harness。

来源：lecture-09-definition-of-done.md；lecture-10-end-to-end-testing.md；lecture-11-harness-observability.md
操作卡片

三层终止校验

静态检查、运行时行为、系统级端到端都通过，才允许声明完成。核心功能没验证通过前，不进入重构或优化。

完成定义 端到端 可观测性
错误消息标准
  • 说明什么出了问题。
  • 说明为什么这是问题。
  • 给出 agent 下一步应修改的位置或路径。
  • 把重复审查意见升级成自动检查。
双层观测

运行时 + 过程

运行时信号回答系统做了什么；冲刺合同、评分标准和任务轨迹回答为什么这样做以及为什么接受或拒绝。

冲刺合同范围、验证标准、排除项先写清。
静态检查lint、typecheck、格式和基础构建。
行为验证单元、集成、启动和关键路径。
端到端跨组件用户流程和真实边界。
评分评估按证据维度判断质量。
反馈闭环失败信息指向修复位置。
讲义详情：完成定义、端到端测试与可观测性

第九讲. 防止 agent 提前宣告完成

痛点

agent 常把代码写完、单元测试绿了或看起来合理误判为完成，跳过系统级验证。

概念

过早完成声明、置信度校准偏差、终止标准、验证-确认双闸门、运行时反馈信号、完成优先级约束。

做法

外部化终止判定；构建三层校验；设计带修复指导的错误消息；捕获启动、关键路径、副作用和清理信号。

案例

课程材料中的密码重置任务表明，单测通过仍可能漏掉邮件服务配置、迁移一致性和完整流程问题。

第十讲. 跑通完整流程才算真正验证

痛点

单元测试隔离依赖，会系统性漏掉接口不匹配、状态传播、资源生命周期和环境依赖问题。

概念

组件边界缺陷、测试充分性梯度、架构边界执行规则、审查反馈提升、面向 agent 的错误消息。

做法

把端到端层纳入完成条件；把架构规则转成 lint/test；错误消息写清问题、原因和修复路径；审查反馈持续提升 harness。

案例

课程材料中的 Electron 导出功能显示，各层单元测试都通过时，端到端测试仍能发现路径、IPC、句柄和权限等边界缺陷。

练习

选一个跨三组件修改，先跑单元测试再跑端到端测试，分析额外发现的缺陷属于哪类交互问题。

来源

raw/ai-workflows/lecture-10-end-to-end-testing.md

第十一讲. 让 agent 的运行过程可观测

痛点

没有仪表盘时，agent 无法区分正确和看似正确，重试变成盲猜，交接也退化为重新诊断。

概念

运行时可观测性、过程可观测性、任务轨迹、冲刺合同、评估评分标准、双层可观测性。

做法

自动采集应用生命周期、关键路径、数据流、资源利用和异常；任务前写冲刺合同；用评分标准统一验收。

案例

课程材料中的三 agent 架构把 planner、generator、evaluator 分工并记录成本、阶段和 QA 反馈，用证据驱动下一轮修复。

练习

审查当前 harness 的系统层和过程层观测缺口；为真实任务写冲刺合同并比较有无合同时的质量差异。

清洁交接：L12

长期可靠性来自每次会话结束时的状态质量。清洁状态不是善后，而是完成定义的一部分。

来源：lecture-12-clean-session-handoff.md
操作卡片

会话完成 = 任务验证 + 清洁状态

构建通过、测试通过、进度记录、临时工件清理、标准启动路径可用。缺一个，下个会话都会替上个会话付费。

清洁状态 会话完整性 质量文档
双模式清理
  • 即时清理：每个会话结束前清理本次创建的临时状态。
  • 定期清理：周期扫描结构性退化和质量评分。
  • 幂等清理：同一清理脚本重复执行仍安全。
  • 定期简化：禁用某个 harness 组件跑基准，验证是否还必要。
风险雷达

“以后再清理”通常就是永远不清理

agent 会复制已有模式。仓库越乱，新会话越倾向在混乱上继续堆代码，重建时间和缺陷逃逸都会上升。

构建标准 build 或 check 命令通过。
测试已有测试和新增验证不被破坏。
记录进度、功能清单、阻塞和下一步写回。
清理调试代码、临时文件、过期 TODO 移除或归档。
启动新会话按标准路径可继续工作。
讲义详情：清洁状态与会话交接

第十二讲. 每次会话结束前都做好交接

痛点

会话结束时测试红、构建坏、临时文件和进度缺失，会让下个会话先花时间收拾状态而非继续交付。

概念

清洁状态、会话完整性、质量文档、清理循环、harness 简化、幂等清理。

做法

把退出检查清单写入 harness；每次即时清理，每周定期清理；维护质量文档；定期移除不再必要的约束。

案例

课程材料中的 12 周对比显示，有清洁策略的项目在构建通过率、测试通过率和新会话启动时间上更稳定。

练习

为代码库设计会话退出检查表，连续 5 个会话应用，记录构建、测试、进度、工件和启动路径的违反次数。
