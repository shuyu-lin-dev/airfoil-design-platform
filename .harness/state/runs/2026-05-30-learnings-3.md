# 学习记录 — 2026-05-30 已知问题修复 session

## 违规：tasks.yaml 事后批量补更

**触发场景**：用户要求将 PROGRESS.md 已知问题写成任务并用 harness 控制完成。进入 Plan 模式→写计划→获批→直接开始实现 T016→T017→T018→三个任务全部 commit 后才将任务条目写入 tasks.yaml 并标记 passing。

**正确顺序**：

1. 先往 tasks.yaml **新增** T016/T017/T018 条目（status=`active`），提交"新建任务条目"
2. 按 WIP=1 逐任务执行，每个任务完成后**立即**更新该任务 status→`passing` + evidence
3. 不等到全部完成再批量写入

**实际顺序**：

1. Plan 获批后直接写代码
2. T016→T017→T018 三连 commit（代码变更）
3. 最后才往 tasks.yaml 写入 T016/T017/T018（一次性，全部 `passing`）
4. 最后才更新 PROGRESS.md、telemetry.yaml

**根因**：

tasks.yaml 被当成"事后文档"而非"事前任务模型"。plan 中虽然描述了任务拆分，但没有把 plan 中的任务定义同步到 tasks.yaml 这个事实源中。

**与过往 pitfall 的异同**：

- 相同：都违反了"禁止批量补更 tracking"的约束
- 不同：此次触因不是"验证=完成"，而是"plan 获批=可以直接写代码，tracking 后续补"

**纠正措施**：

- Plan 获批后的第一个动作必须是：将 plan 中定义的任务条目添加到 tasks.yaml（status=`active` 或 `not_started`），并提交。此后才能开始写代码。
- tasks.yaml 是执行的前置依赖，不是执行后的归档。没有条目就不能动代码。
